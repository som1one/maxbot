# Bugfix Requirements Document

## Introduction

В проде Telegram-бота `maxbot` (Python, FastAPI backend в `app/`, отдельный воркер `bot/worker.py`, SQLAlchemy + Alembic, SMTP-уведомления через `app/emailer.py`) обнаружены четыре связанных критических дефекта, приводящих к потере клиентских заявок и неконтролируемой публичной поверхности атаки:

1. **Admin-уведомления молча не доходят** — `ADMIN_CHAT_ID` невалиден (формат/тип/user_id вместо chat_id), бот не валидирует значение при старте и проглатывает ошибки `sendMessage`.
2. **SMTP заблокирован у Timeweb Cloud (с 25 мая)** — исходящие письма с уведомлениями о заявках падают, ретраев и альтернативного канала нет, потеря тихая.
3. **Заявки не пишутся в БД** — обработчик в `bot/worker.py` принимает данные и не вызывает `session.add(...) / commit()`, при том что модели в `app/models.py` и миграции в `alembic/versions/` существуют и применены.
4. **Backend (FastAPI) висит на публичном IP без аутентификации/файрвола** — приложение биндится на `0.0.0.0`, легитимного внешнего вызова нет, эндпоинты (включая админские) открыты.

Вместе эти дефекты приводят к полной потере заявки: данные не сохраняются ни в БД (баг 3), ни по почте (баг 2), ни в Telegram админу (баг 1), а сервис при этом доступен из интернета без защиты (баг 4). Реальная заявка («Юлия Иванова, 7 мая, носки из Китая, +79163632551, info@wsgarant.ru») была потеряна по этой причине; восстановление этой заявки выполняется операционно вне кода и **в скоуп данного фикса не входит**.

Скоуп фикса — четыре баг-кондишена (C1-C4) ниже. Часть фиксов кодовая (валидация, персист, retry-задача, аутентификация эндпоинтов), часть — инфраструктурная (bind / firewall / reverse proxy на хосте Timeweb Cloud); требования покрывают и то, и другое.

## Bug Analysis

### Current Behavior (Defect)

Текущее наблюдаемое поведение в проде. Условия C1-C4 — это бинарные предикаты на состоянии системы / входе обработчика, при истинности которых проявляется соответствующий дефект.

**C1 — Admin-уведомления (ADMIN_CHAT_ID невалиден / sendMessage падает молча):**

1.1 WHEN бот стартует с `ADMIN_CHAT_ID`, который отсутствует, не парсится как целочисленный telegram chat_id, либо является user_id, к которому у бота нет доступа на запись, THEN the system стартует без ошибок и без предупреждений, как будто значение валидно.

1.2 WHEN бот пытается отправить admin-уведомление через `sendMessage` и Telegram API возвращает ошибку (например `Bad Request: chat not found`, `Forbidden: bot was blocked`, таймаут), THEN the system проглатывает исключение, не пишет ошибку в лог с уровнем ERROR и продолжает работу как будто уведомление доставлено, при этом отсутствует health-check / `/ping`-команда, которой админ мог бы проверить доставляемость.

**C2 — SMTP заблокирован у Timeweb Cloud, fallback отсутствует:**

1.3 WHEN воркер пытается отправить письмо о заявке через настроенный SMTP-провайдер и соединение/отправка падает (block по портам 25/465/587 со стороны хостера, таймаут, ошибка аутентификации), THEN the system теряет письмо без ретраев, без записи об ошибке доставки в БД и без переключения на альтернативный канал.

1.4 WHEN SMTP-вызов выполняется внутри хендлера приёма заявки синхронно, THEN the system блокирует основной поток приёма заявки на время сетевой ошибки/таймаута SMTP и при падении SMTP не оставляет персистентного следа о принятой заявке.

**C3 — Заявки не пишутся в БД (нет ORM-вызовов в воркере):**

1.5 WHEN клиент отправляет заполненную заявку (имя, дата, описание, телефон, email) и хендлер в `bot/worker.py` валидно её распарсил, THEN the system не вызывает `session.add(...) / session.commit()` ни на одной из существующих моделей в `app/models.py`, и в таблице заявок (orders/requests) не появляется новая строка.

1.6 WHEN отправка уведомлений (Telegram/SMTP) после приёма заявки падает по любой причине, THEN the system не имеет персистентного следа заявки, по которому её можно было бы повторно обработать или восстановить, потому что запись в БД не создавалась изначально.

**C4 — Backend открыт на публичном IP без аутентификации/файрвола:**

1.7 WHEN FastAPI-приложение запускается в проде, THEN the system биндится на `0.0.0.0` и слушает на публичном IP сервера Timeweb Cloud, без ограничения по IP / security group / файрволу на уровне хоста или контейнера.

1.8 WHEN произвольный внешний клиент обращается к эндпоинтам backend (включая административные), THEN the system отвечает без обязательной аутентификации и без авторизации, при этом легитимных внешних вызовов у backend нет (он используется только ботом/админом).

### Expected Behavior (Correct)

Поведение, которое должно стать истинным после фикса, в той же нумерации C(X), что и дефекты выше.

**C1 — Admin-уведомления:**

2.1 WHEN бот стартует, THEN the system SHALL валидировать `ADMIN_CHAT_ID` как целочисленный Telegram chat_id (включая отрицательные id для групп/каналов), и при невалидном/пустом значении SHALL отказаться стартовать с понятной ошибкой в логе уровня ERROR (или, для опциональных режимов, стартовать с явным WARNING и отключённым admin-каналом, без молчаливой деградации).

2.2 WHEN отправка admin-уведомления через Telegram API завершается ошибкой (любой 4xx/5xx, таймаут, `Forbidden`, `chat not found`), THEN the system SHALL логировать ошибку на уровне ERROR с телом ответа Telegram, SHALL не считать уведомление доставленным, и SHALL предоставлять администратору проверяемый health-check (например, команду `/ping` боту или эндпоинт/CLI), который явно сообщает «admin-канал работает / не работает».

**C2 — SMTP / альтернативный канал:**

2.3 WHEN заявка принята, THEN the system SHALL гарантировать доставку уведомления как минимум по одному каналу — Telegram админу И/ИЛИ персистентная запись в БД — независимо от состояния SMTP, и SHALL не считать заявку «потерянной» при падении SMTP.

2.4 WHEN отправка письма через SMTP падает, THEN the system SHALL выполнять отправку как best-effort вне основного потока приёма заявки (фоновая задача / отдельный воркер) с ограниченным числом ретраев и экспоненциальной задержкой, SHALL логировать каждую неуспешную попытку на уровне WARNING/ERROR, SHALL помечать в БД для соответствующей заявки факт «email не доставлен» (например, флаг `email_sent=false` и/или `last_email_error`), и SHALL не блокировать ответ клиенту в боте.

**C3 — Персист заявки в БД:**

2.5 WHEN хендлер заявки в `bot/worker.py` получает валидно распарсенные данные, THEN the system SHALL атомарно сохранять заявку в БД через SQLAlchemy-сессию (`session.add(...) / session.commit()`) на существующей модели заявок из `app/models.py`, ДО отправки внешних уведомлений (Telegram/SMTP), и при ошибке коммита SHALL делать `rollback` и сообщать клиенту понятную ошибку.

2.6 WHEN внешнее уведомление (Telegram админу или email) после успешного коммита заявки не доставлено, THEN the system SHALL сохранять у записи флаг `notified=false` (и/или раздельно `admin_notified`, `email_sent`) с временем последней попытки и текстом ошибки, и SHALL предоставлять retry-механизм (фоновая задача / периодический воркер), который дослыает уведомления для записей с `notified=false` без создания дубликатов записей.

**C4 — Закрытие backend от публичного доступа:**

2.7 WHEN FastAPI-приложение запускается в проде, THEN the system SHALL быть недоступным напрямую с публичного IP — реализовано хотя бы одним из способов: bind на `127.0.0.1` (или внутреннюю docker-сеть) с reverse proxy (nginx/Traefik) перед ним, ИЛИ ограничение на уровне файрвола/security group хостера Timeweb Cloud, ИЛИ комбинацией; способ выбора фиксируется в design и отражается в `docker-compose.prod.yml` / `deploy.sh` / `DEPLOYMENT.md`.

2.8 WHEN произвольный внешний клиент (или внутренний клиент, не являющийся ботом/админом) обращается к административным эндпоинтам backend, THEN the system SHALL требовать обязательную аутентификацию (токен в заголовке / API key, реализованный в `app/security.py` или эквивалентном модуле), SHALL отклонять неаутентифицированные запросы кодом 401/403, и SHALL логировать факт неаутентифицированного запроса.

### Unchanged Behavior (Regression Prevention)

Существующее поведение, которое фикс не должен ломать. Это входы, для которых `NOT C(X)`, и преобразования, не относящиеся к перечисленным C1-C4.

**Приём сообщений и UX бота (вне сценариев баг-кондишенов):**

3.1 WHEN клиент общается с ботом по существующему диалогу заявки (валидные шаги, валидные ответы), THEN the system SHALL CONTINUE TO принимать и парсить сообщения с тем же набором полей (имя, дата, описание, телефон, email) и в том же порядке шагов, что и до фикса.

3.2 WHEN клиент успешно завершает заполнение заявки и все каналы уведомлений (Telegram админу, БД, SMTP) работают, THEN the system SHALL CONTINUE TO отвечать клиенту тем же финальным сообщением/подтверждением, что и до фикса.

**Формат и содержимое admin-уведомлений:**

3.3 WHEN admin-уведомление о заявке успешно отправляется в Telegram (валидный `ADMIN_CHAT_ID`, доступный чат), THEN the system SHALL CONTINUE TO формировать сообщение с тем же набором полей и форматированием, что и до фикса.

3.4 WHEN email-уведомление о заявке успешно отправляется через SMTP (когда SMTP доступен), THEN the system SHALL CONTINUE TO использовать тот же шаблон письма (тема, тело, адресаты), что и до фикса.

**Схема БД и миграции:**

3.5 WHEN применяются существующие миграции из `alembic/versions/` (`001_initial_migration`, `daaddf629e95_init`, `dea93c3b67f9_init`, `0d0fcf2a1281_add_order_date_field`), THEN the system SHALL CONTINUE TO применять их в том же порядке и приводить БД к тому же состоянию, что и до фикса; новые поля (`notified`, `admin_notified`, `email_sent`, `last_email_error` и т.п.) SHALL добавляться отдельной новой миграцией поверх существующей цепочки, без изменения уже применённых ревизий.

3.6 WHEN читаются существующие записи заявок (если такие имеются) после фикса, THEN the system SHALL CONTINUE TO корректно их десериализовать через ORM-модели, без падений из-за новых обязательных not-null полей (новые поля SHALL иметь дефолт или быть nullable).

**Конфигурация и деплой:**

3.7 WHEN деплой выполняется через существующие `docker-compose.yml` / `docker-compose.prod.yml` / `deploy.sh`, THEN the system SHALL CONTINUE TO стартовать те же сервисы (bot worker, backend, db, миграции) и читать те же переменные окружения из `.env` (с возможным расширением списка переменных, документированных в `.env.example` и `DEPLOYMENT.md`).

3.8 WHEN в окружении заданы валидный `ADMIN_CHAT_ID` и работающий SMTP, THEN the system SHALL CONTINUE TO доставлять уведомления по обоим каналам в штатном режиме, без необязательных ретраев и без посторонних предупреждений в логах.

**Backend API для легитимных вызовов:**

3.9 WHEN backend вызывается легитимным внутренним клиентом (бот / админ / reverse proxy), с валидной аутентификацией там, где она требуется, THEN the system SHALL CONTINUE TO возвращать те же контракты ответов (схемы из `app/schemas.py`, коды статусов), что и до фикса.

3.10 WHEN backend стартует локально для разработки (вне прод-деплоя), THEN the system SHALL CONTINUE TO быть доступным разработчику на `localhost` для отладки, без обязательного включения прод-файрвола/прод-bind на dev-машине.
